# Поточний стан роботи

## Базові відомості

- Повна тема роботи: **«Розробка архітектури АІНС на основі великих мовних моделей для реалізації діалогового агента з функціями репетитора»**.
- Тип роботи: **магістерська курсова робота**, не диплом і не повна магістерська кваліфікаційна робота.
- АІНС: **адаптивна інтелектуальна навчальна система**.
- Предметна область прототипу: **навчання базового JavaScript**.
- Практичний vertical slice: **генерація вправи -> написання/редагування рішення -> запуск тестів -> адаптивний фідбек у стилі репетитора**.
- Основний принцип: **architecture-first**. Розділ 3 має бути центральним і сильнішим за розділ 4.

## Межі архітектури та реалізації

- У документі буде описано повну архітектуру АІНС:
  - модулі та компоненти;
  - модель студента;
  - модель знань;
  - модель репетитора;
  - пам'ять агента;
  - потоки даних і цикл прийняття рішень.
- У практичній частині достатньо одного працюючого vertical slice.
- На момент початку роботи в репозиторії відсутній прикладний код; є лише шаблонні `.docx` у `assets/`.
- Щоб розділ 4 не суперечив реальному стану репозиторію, буде створено мінімальний прототип, який покриває:
  - Chat UI;
  - ExerciseGenerator;
  - TutorPolicy;
  - ClientTestRunner;
  - FeedbackEvaluator;
  - SessionMemory.
- Концептуально буде описано ширшу архітектуру з buffer/session memory, summary memory, vector/profile memory, knowledge memory, навіть якщо не всі типи пам'яті будуть реалізовані у коді.

## Реальний стан прототипу після етапу 2

- У репозиторії створено мінімальний vertical slice в `prototype/`.
- Реальний стек практичної частини:
  - статичний frontend на HTML/CSS/JavaScript без окремого бекенду;
  - браузерні ES modules;
  - виклики до локального Ollama через HTTP API;
  - виконання студентського коду в `Web Worker`;
  - локальні unit-тести через `node --test`.
- Реально реалізовано:
  - `Chat UI` як інтерфейс із журналом взаємодій;
  - `ExerciseGenerator` з JSON-підказкою до Ollama та fallback-вправами;
  - `TutorPolicy` з матрицею рішень за `attemptsCount`, типом помилки та запитом на пояснення;
  - `ClientTestRunner` з worker-based sandbox і таймаутом;
  - `FeedbackEvaluator` з LLM/fallback фідбеком;
  - `SessionMemory` з `UserState`, історією помилок і transcript tail.
- Концептуально описуватиметься, але в коді поки не реалізовано:
  - RAG і зовнішнє retrieval;
  - vector/profile memory між сесіями;
  - summary memory;
  - knowledge graph або окрема база навчального контенту;
  - багатокрокове планування агента з інструментами поза поточним vertical slice;
  - LMS-інтеграція та педагогічна аналітика.
- Локальний `ollama` binary в поточному середовищі не встановлено, тому код підтримує fallback-режим. Це треба чесно відобразити в розділі 4.

## Обраний runtime та технічні обмеження

- Основний LLM runtime: **локальний Ollama**.
- Пріоритетна модель: **`llama3.1:8b`** або сумісна локальна open-source модель.
- Виконання користувацького JavaScript-коду: **лише на клієнті у Web Worker з таймаутом**.
- Заборонено server-side `eval`.
- Для взаємодії з LLM потрібні:
  - структурований JSON-вивід;
  - retry/fallback, якщо модель повертає невалідний JSON;
  - обмеження на пряме розкриття повного розв'язку.

## Що вже з'ясовано з шаблонів

- `assets/Зміст курсового проєкту.docx` задає рекомендований зміст і окремо підкреслює розмежування між архітектурою та програмною реалізацією.
- `assets/маг робота як шаблон оформлення.docx` є прикладом візуальної структури магістерської роботи КНУ:
  - формат сторінки: Letter;
  - поля приблизно: ліве 20 мм, праве 10 мм, верхнє 20 мм, нижнє 20 мм;
  - базовий текст: Times New Roman 14 pt, міжрядковий інтервал 1.5;
  - титульні та службові сторінки містять значну кількість змінних реквізитів, які не можна вигадувати.
