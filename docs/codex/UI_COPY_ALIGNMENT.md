# UI Copy Alignment

## Джерела термінології

Термінологію звірено з:

- template DOCX
- фінальним submission-ready DOCX `v6`
- формулюваннями з розділів про:
  - мінімальний практичний сценарій
  - модель стану студента
  - режим пояснення
  - генерацію вправи, запуск тестів і адаптивний фідбек

## Назви, які потрібно зберігати

- `вправа`
- `розв'язок`
- `перевірочні тести`
- `адаптивний фідбек`
- `режим пояснення`
- `агент-репетитор`
- `студент`
- `репетитор`
- `пам'ять сесії`
- `модель стану студента`
- `локальна модель`
- `резервний режим`
- `JSON-експорт сесії`

## Назви, які можна покращувати без зміни змісту

- `Session Memory` -> `Пам'ять сесії`
- `Режим LLM` -> `Статус локальної моделі`
- сирі action tokens:
  - `idle` -> `очікування`
  - `exercise_generated` -> `вправу підготовлено`
  - `concept_explanation` -> `режим пояснення`
- `starter code` -> `початковий код`
- загальні dev-пояснення на кшталт `vertical slice` -> нейтральний опис практичного сценарію

## Назви, які не можна змінювати

- `режим пояснення`
- `модель стану студента`
- `резервний режим`
- `локальна модель Ollama`
- `перевірочні тести`
- `вправа`
- `студент`
- `репетитор`

## Слова й тони, яких слід уникати

- `Learn`
- `Challenge`
- `Coach mode`
- `Memory graph`
- випадкові англомовні dev/demo маркери без потреби
- product-marketing copy, який суперечить академічному тону курсової

## Практичне правило

- Якщо в coursework термін уже має архітектурне або педагогічне значення, UI лише покращує подачу, але не вигадує нову назву.
- Якщо поточний текст є службовим або сирим токеном, його можна локалізувати й нормалізувати до академічно-нейтральної форми.
