# UI/UX Audit

## Scope

- Прочитано template DOCX, фінальний submission-ready DOCX, `prototype/README.md` і весь код у `prototype/`.
- Проведено pre-rewrite browser audit для desktop і mobile.
- Для audit/visual direction використано `ui-ux-pro-max`.

## Вихідні проблеми

1. Екран не мав достатньо чіткої ієрархії.
   Чотири панелі виглядали як рівнозначні технічні блоки. Guided tutoring flow читався слабо: було неочевидно, з чого починати і де зараз головна дія.

2. Групи дій були технічно працездатні, але візуально слабкі.
   `Generate`, `Run tests`, `Explain`, `Export`, `Refresh status` не формували виразної primary/secondary hierarchy. На вузькому екрані кнопки вже переносилися, але відчуття злипання і випадкових переносів лишалося.

3. Exercise panel був монолітним.
   Умова, код, тести, поняття й metadata рендерилися в одному blob-блоці. Це ускладнювало сканування й робило вправу схожою на сирий debug output.

4. Test results були надто стислі.
   Користувач бачив лише загальний summary і максимум кілька назв провалених тестів. Не вистачало чітких pass/fail rows, окремого summary і зрозумілого наступного кроку.

5. Feedback panel змішував різні ролі.
   System status, pending state, результат перевірки і власне репетиторський фідбек були злиті в один візуальний простір. Через це панель читалася як internal tool, а не як навчальний продукт.

6. Transcript виглядав як список `div`.
   Повідомлення студента і репетитора не мали достатньої відмінності, довші відповіді погано сканувалися, pending-state не виглядав як частина повноцінного діалогу.

7. Статус локальної моделі мав dev-сприйняття.
   Бракувало окремих станів `готова / перевіряється / недоступна / резервний режим`. Plain-text статус не давав відчуття стабільного продукту.

8. Empty states були занадто загальними.
   Стани `немає вправи`, `немає результатів`, `немає пояснення`, `діалог не розпочато` подавалися як сухий текст без контексту й без guidance.

9. Waiting feedback уже існував, але не був достатньо цілісним.
   Було видно, що робота над loading states уже починалася, але вони виглядали як доповнення поверх demo UI, а не як частина єдиної дизайн-системи.

10. Copy частково конфліктував із тоном курсової.
    `Session Memory`, `vertical slice`, сирі action tokens на кшталт `idle` і загальний demo-tone не підтримували термінологічну узгодженість із submission-ready DOCX.

## Висновок

- До rewrite інтерфейс був функціональним, але сприймався як internal prototype.
- Основний дефіцит був не в логіці, а в presentation layer: spacing, hierarchy, states, copy alignment і відсутність цілісного visual direction.

## Сигнали з `ui-ux-pro-max`, які були взяті в роботу

- Показувати explicit loading feedback для всіх операцій довше за 300 мс.
- Забезпечити достатній touch spacing між сусідніми кнопками.
- Утримувати послідовну heading hierarchy.
- Використати стабільну типографічну шкалу й виразні status states.
- Не лишати UI в підвішеному стані без spinner/skeleton/disabled feedback.
